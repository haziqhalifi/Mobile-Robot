# Mobile Robot Project Guide

## 1. What this repository currently is

This is a **PlatformIO Arduino firmware project**. PlatformIO compiles the C/C++ code into firmware and uploads it to a microcontroller.

At present, the executable application is only PlatformIO's starter example: it adds `2 + 3` once and then does nothing. The repository also contains libraries for:

- a four-wheel mecanum robot motor controller;
- an NEC-style infrared remote receiver;
- hobby servo motors; and
- Adafruit NeoPixel addressable LEDs.

None of those robot libraries are used by `src/main.cpp` yet.

### Important hardware mismatch

`platformio.ini` selects an **Arduino Uno** (`ATmega328P`, AVR, 16 MHz). However, comments in `MecanumCar_v2` and `ir` say those libraries were written for a **Raspberry Pi Pico**, and the IR decoder says its timing assumes a 125 MHz clock.

This means:

- the current starter program can build for an Uno;
- NeoPixel and Servo support the Uno;
- the custom motor code may compile on an Uno because it uses standard Arduino calls, but its pin choices, electrical connection, and target motor-controller board must be checked;
- the IR decoder's timing is not reliable on an Uno without modification; and
- if the physical robot uses a Pico, `platformio.ini` must be changed to a Pico environment before developing the real firmware.

Do not connect motors or motor power directly to microcontroller GPIO pins. The code expects a separate controller at I2C address `0x30`.

## 3. Root files and generated folders

### `platformio.ini`

The main PlatformIO configuration:

```ini
[env:uno]
platform = atmelavr
board = uno
framework = arduino
```

- `[env:uno]` names the build environment `uno`.
- `platform = atmelavr` selects the AVR development platform/toolchain.
- `board = uno` selects the Arduino Uno and its ATmega328P settings.
- `framework = arduino` supplies `setup()`, `loop()`, GPIO, timing, and other Arduino APIs.

This file is the source of truth for board settings. Edit it instead of generated VS Code configuration.

### `.gitignore`

Prevents generated or machine-specific files from being committed:

- `.pio` — compiled objects and firmware;
- `.vscode` database and IntelliSense files;
- `.vscode/launch.json` — generated debug setup;
- `.vscode/ipch` — IntelliSense precompiled headers.

### `.pio/`

Generated by PlatformIO. It contains downloaded/build metadata and files such as:

- object files;
- static libraries;
- `firmware.elf`;
- `firmware.hex`;
- dependency information.

It can be regenerated with a build and normally should not be edited or committed.

### `.vscode/`

Editor support generated or used by PlatformIO:

- `c_cpp_properties.json` — include paths, compiler, C/C++ standard, and Uno preprocessor definitions used by IntelliSense. It is auto-generated.
- `extensions.json` — recommends the PlatformIO IDE extension and discourages the overlapping C/C++ extension pack.
- `launch.json` — generated PlatformIO debug configurations pointing to `.pio/build/uno/firmware.elf`.

These files affect the editor, not the robot's runtime behavior.

## 4. Application code

### `src/main.cpp`

This is the firmware entry point and the only application source file.

```cpp
#include <Arduino.h>
```

Provides Arduino types and functions.

```cpp
int myFunction(int, int);
```

Declares a function before it is called. In C++, the compiler must know a function's signature before use.

```cpp
void setup() {
  int result = myFunction(2, 3);
}
```

`setup()` runs once after reset. It calculates `5`, stores it in a local variable, and then discards it. There is no visible output.

```cpp
void loop() {
}
```

`loop()` runs repeatedly forever, but is empty.

```cpp
int myFunction(int x, int y) {
  return x + y;
}
```

An example function that returns the sum of two integers.

To turn this into robot firmware, `main.cpp` will eventually need to create the relevant device objects, configure pins in `setup()`, and read sensors/control movement in `loop()`.

## 8. Mecanum motor library

### `lib/MecanumCar_v2/MecanumCar_v2.h`

Declares the `mecanumCar` class and four global speed values:

- `speed_Upper_L`
- `speed_Lower_L`
- `speed_Upper_R`
- `speed_Lower_R`

Each speed is an 8-bit value from `0` to `255`.

The constructor accepts two GPIO numbers:

```cpp
mecanumCar(int _sda, int _scl);
```

They are used as software-driven SDA and SCL lines. This code does **not** use Arduino's `Wire` library.

Public movement methods:

| Method | Intended motion |
|---|---|
| `Init()` | Wait, turn both LEDs off, and stop all motors |
| `Stop()` | Stop all four motors |
| `Advance()` | Move forward |
| `Back()` | Move backward |
| `Turn_Left()` | Rotate left in place |
| `Turn_Right()` | Rotate right in place |
| `L_Move()` | Strafe left |
| `R_Move()` | Strafe right |
| `LU_Move()` | Move diagonally left-forward |
| `LD_Move()` | Move diagonally left-backward |
| `RU_Move()` | Move diagonally right-forward |
| `RD_Move()` | Move diagonally right-backward |
| `drift_left()` | Drive only the lower motors in opposing directions |
| `drift_right()` | Opposite lower-motor drift pattern |

Low-level methods control individual motors, LEDs, PWM registers, and the software I2C bus. They are public even though application code would normally use only the movement and LED methods.

### `lib/MecanumCar_v2/MecanumCar_v2.cpp`

Implements the class.

#### Speed defaults

All motors start at PWM `80` out of `255`, about 31% of the numeric range. The variables are global, so application code can change each wheel independently.

#### Motor naming

The library treats the drivetrain as:

```text
front/upper left  (UL)       front/upper right (UR)
rear/lower left  (LL)       rear/lower right  (LR)
```

“Upper” and “lower” appear to mean the front and rear positions in the original robot diagram.

#### Controller register mapping

Each motor uses two PWM channels. One channel is set to the requested speed and the other to zero to choose direction.

| Motor | Channel pair |
|---|---:|
| Upper right | 1, 2 |
| Upper left | 3, 4 |
| Lower right | 5, 6 |
| Lower left | 7, 8 |
| Left LED / P54 | register `0x09` |
| Right LED / P55 | register `0x0A` |

Every register write sends:

```text
START → 0x60 → register address → data byte → STOP
```

`0x60` is `0x30 << 1`, meaning a 7-bit device address of `0x30` with the write bit.

#### Software I2C functions

- `IIC_Start()` creates a start condition.
- `IIC_Stop()` creates a stop condition.
- `IIC_SendByte()` sends eight bits, most-significant bit first.
- `IIC_RecvACK()` samples the device's acknowledgment.
- `IIC_SendACK()` and `IIC_RecvByte()` exist but are not used by current writes.

The code ignores the result of `IIC_RecvACK()`, so a disconnected or unresponsive controller is not reported to the application.

#### Movement logic

A mecanum robot moves by assigning a direction and speed to each wheel. For example:

- forward: all wheels use the same direction;
- rotation: left and right sides use opposite directions;
- strafing: diagonal wheel pairs use opposite directions;
- diagonal travel: two wheels move while the other two stop.

Actual direction depends on motor wiring. If a motor is mounted or wired oppositely, a named movement can produce the wrong physical motion.

#### Risks and limitations

- It bit-bangs I2C without open-drain behavior; normal I2C devices expect pull-up resistors and lines that are released rather than actively driven high.
- ACK failures are ignored.
- Timing consists of fixed 1 µs delays and may vary by board.
- The class is named `mecanumCar` rather than the usual C++ `MecanumCar`.
- Speeds are global, not encapsulated in each car object.
- Header comments are mojibake because some original Chinese text was read/saved with the wrong character encoding.

## 9. Infrared remote library

### `lib/ir/ir.h`

Declares class `IR`:

- `IR(int p)` stores the receiver GPIO and configures it as input.
- `IRStart()` detects the approximately 9 ms low leading pulse.
- `getByte()` reads eight data bits.
- `getKey()` reads and validates a complete four-byte command.

### `lib/ir/ir.cpp`

Implements a blocking NEC-like IR decoder.

The expected frame is:

```text
9 ms leading LOW
4.5 ms leading HIGH
address
inverted address
command
inverted command
```

`getKey()` returns:

- the command byte (`0`–`255`) when both inverse-byte checks pass; or
- `-1` when no valid frame is found.

Validation uses:

```cpp
key[0] + key[1] == 0xff
key[2] + key[3] == 0xff
```

Important limitations:

- It busy-waits inside `while` loops, so the rest of the robot cannot run concurrently while a frame is being decoded.
- Some loops have no timeout and can hang if the signal becomes stuck.
- The source explicitly says its timing is for a 125 MHz Raspberry Pi Pico.
- It does not handle NEC repeat frames for a held button.
- Comments contain mojibake caused by an encoding mismatch.

## 10. Servo library

This is Arduino Servo library version **1.1.2**.

### Metadata and documentation

- `lib/Servo/library.properties` — library name, version, author, category, URL, and supported architectures.
- `lib/Servo/README.adoc` — purpose and LGPL license information.
- `lib/Servo/keywords.txt` — syntax-highlighting keywords for Arduino IDE.

### Public API

- `lib/Servo/src/Servo.h` — declares the `Servo` class and methods such as `attach()`, `detach()`, `write()`, `writeMicroseconds()`, `read()`, and `attached()`.

### Architecture implementations

Only the implementation matching the selected board is compiled:

| Folder | Target family |
|---|---|
| `src/avr/` | Arduino Uno/Mega and other AVR boards |
| `src/megaavr/` | Newer megaAVR boards |
| `src/sam/` | Arduino Due family |
| `src/samd/` | SAMD boards such as Zero/MKR |
| `src/nrf52/` | Nordic nRF52 boards |
| `src/stm32f4/` | STM32F4 boards |
| `src/mbed/` | Arduino mbed-based boards |

Within each architecture folder:

- `Servo.cpp` implements pulse generation and timer interrupt behavior.
- `ServoTimers.h` selects timers and architecture-specific definitions.

For the current Uno configuration, `src/avr/Servo.cpp` and `src/avr/ServoTimers.h` are relevant. Servo timing consumes hardware timer resources, so it can conflict with PWM on certain pins or other timer-based libraries.

### Examples

- `examples/Knob/Knob.ino` — reads an analog potentiometer and maps it to servo position.
- `examples/Sweep/Sweep.ino` — repeatedly sweeps a servo through an angle range.

Examples are references and are not part of the project firmware build.

## 11. Adafruit NeoPixel library

This is Adafruit NeoPixel library version **1.9.0** for one-wire addressable RGB/RGBW pixels.

### Main files

- `Adafruit_NeoPixel.h` — class declarations, pixel types, color-order flags, and public API.
- `Adafruit_NeoPixel.cpp` — shared implementation, including pixel buffers, brightness, colors, and `show()`.
- `library.properties` — Arduino library metadata and version.
- `keywords.txt` — Arduino IDE syntax highlighting.
- `README.md` — installation, supported chips, API summary, and usage.
- `COPYING` — LGPL license text.
- `CONTRIBUTING.md` — upstream contribution rules.

### Architecture-specific files

- `esp.c` — ESP32-specific output support.
- `esp8266.c` — ESP8266-specific output support.
- `kendyte_k210.c` — Kendryte K210 support.
- `rp2040.c` — RP2040/Pico support.
- `rp2040_pio.h` — RP2040 PIO program declarations/data.

For an Arduino Uno, the AVR-specific timing is primarily selected inside `Adafruit_NeoPixel.cpp`; the files above for other processors are not used.

`show()` must produce very precise signal timing. On many boards it temporarily disables interrupts, which can delay serial reception, servo timing, or other time-sensitive work when updating long LED strips.

### Examples

- `examples/simple/simple.ino` — minimal fixed-pixel example.
- `examples/simple_new_operator/simple_new_operator.ino` — dynamically allocates the strip object.
- `examples/strandtest/strandtest.ino` — standard color and animation test.
- `examples/strandtest_wheel/strandtest_wheel.ino` — uses a color-wheel helper.
- `examples/RGBWstrandtest/RGBWstrandtest.ino` — tests RGBW pixels.
- `examples/buttoncycler/buttoncycler.ino` — changes animations with a button.
- `examples/StrandtestArduinoBLE/StrandtestArduinoBLE.ino` — BLE-controlled example using ArduinoBLE.
- `examples/StrandtestArduinoBLECallback/StrandtestArduinoBLECallback.ino` — callback-based ArduinoBLE variant.
- `examples/StrandtestBLE/StrandtestBLE.ino` — BLE strand test.
- `examples/StrandtestBLE/BLESerial.h` and `BLESerial.cpp` — helper class used by that BLE example.

The BLE examples require additional hardware and libraries and are not suitable for a standard Uno by themselves.

## 12. What is compiled right now

PlatformIO always compiles `src/main.cpp`. It discovers private libraries when application code includes them.

Because `main.cpp` currently includes only `<Arduino.h>`, the robot, IR, Servo, and NeoPixel libraries are not functionally linked into the application. Their presence in `lib/` does not activate them.

Typical includes would look like:

```cpp
#include <MecanumCar_v2.h>
#include <ir.h>
#include <Servo.h>
#include <Adafruit_NeoPixel.h>
```

Do not add all of them blindly. First confirm the real controller board, pinout, peripherals, and power system.

## 14. Recommended next design

Before implementing behavior, record the actual hardware:

1. Controller board model: Uno or Raspberry Pi Pico.
2. SDA and SCL pins connected to the motor controller.
3. Motor-controller model and confirmation that its 7-bit address is `0x30`.
4. Physical location and direction of each motor.
5. IR receiver GPIO and remote command codes.
6. Servo signal GPIO and safe mechanical angle limits.
7. NeoPixel GPIO, pixel count, color order, and separate power requirements.
8. Common ground arrangement and battery/motor voltage.

Then organize application-specific code approximately as:

```text
include/
├── pins.h
├── robot_config.h
└── remote_codes.h
src/
├── main.cpp
├── drive_controller.cpp
├── remote_controller.cpp
└── status_lights.cpp
test/
└── ...real tests...
```

Keeping hardware assignments and command codes out of `main.cpp` makes the system easier to understand and change.
